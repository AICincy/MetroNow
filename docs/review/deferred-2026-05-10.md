## A. CSS — hardcoded badge colors

**Location.** `web/public/css/atlas-supplement.css`, ~11 sites
across lines 13, 430-435, 440, 444, 463-464, 496-513, 520.

**Audit source.** Web-frontend subagent (Blocker). Sentinel-3
independently verified that the cited lines do carry literal hex
values rather than `var(--token)` references.

**What's there now.** Hex literals for skip-nav text on brand
background, CAGIS confidence badges, route-diff badges, Osmose
badges, and discussion-tab badges:

- `#fff` — skip-nav text over brand-orange
- `#e7e7e7`, `#333`, `#d3f4dc` — confidence-badge variants
- Direct status-badge hexes
- `rd-real` route-diff hex values

**Why this needs discussion, not a sweep.** The right fix is a
new token taxonomy, not piecemeal `#xxx` → `var(--something)`
substitution. Three open questions need answers first:

1. Tier scale or domain scale? `--badge-ok` / `--badge-warn` /
   `--badge-err` is reusable across components but generic;
   `--conf-ok` vs `--rd-real` is precise but doubles the token
   count.
2. How do these interact with the existing dark/light theme
   system in `web/public/index.html:82-99`? Some badges are
   intentionally non-theming (skip-nav must stay readable in
   either theme); others should follow.
3. The skip-nav `color: #fff` over brand-orange is
   intentional high-contrast. Do we leave it as a literal
   anchor or token it as `--brand-contrast`?

**Proposed.** Define the taxonomy in a short design note,
review, then sweep `atlas-supplement.css` in a single PR. The
sweep should be one commit per token to keep diffs reviewable.

## B. Google Fonts SRI hash

**Location.** `web/public/index.html:9`.

**Audit source.** Web-frontend subagent, Info-level.

**What's there now.** The Google Fonts CSS `<link>` carries no
`integrity` attribute, unlike every other unpkg CDN load on the
same page.

**Why it's hard.** Google Fonts CSS varies per request — the
URL includes query parameters (`display=swap`, font subsets)
and the returned CSS itself rotates as fonts are updated.
Pinning a SRI hash today breaks at the next Google Fonts
revision, which can happen silently.

**Options.**

1. **Self-host the font CSS.** Eliminates the SRI problem;
   adds an asset-pipeline maintenance burden the project has
   deliberately avoided so far.
2. **Add a CSP `style-src 'sha256-...'` for the inline `<style>`
   block.** The current page already trusts Google Fonts via
   the helmet allow-list; this would tighten it further.
3. **Document the gap, accept the risk.** Google Fonts is a
   first-party Google service; the threat model for an
   integrity-failure attack against the font CSS is narrow.

**Proposed.** Discuss host-vs-accept with the operator before
acting.
