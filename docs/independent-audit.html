§1Executive summary

The Atlas UI is a faithful super-set of the upstream repository's contracted surfaces. Every endpoint the UI claims to call exists in web/server.js, every defect class it renders matches src/osm/classify.py, and the four service zones it offers are the exact zones declared in src/osm/zones.py. The UI has, however, grown three surfaces — the Discussion board, the Formality & Reports panel, and the Command console — that have no backend persistence in the current repo and live entirely in the browser. These are clearly scoped and harmless, but should be flagged to operators as "client-only" until corresponding endpoints land.

The audit covers four dimensions: (1) feature parity with the repo's UI contract in web/public/index.html; (2) data-model fidelity to src/osm/classify.py and src/osm/review.py; (3) endpoint coverage against web/server.js; and (4) compliance with the OSM-community norms recorded in CLAUDE.md and README.md.

§5Functional checks performed

  1. Page load. MetroNow Atlas.html opens with no console errors; verifier subagent confirmed clean state.
  2. Extras layer. assets/atlas-extras.js registers a single MutationObserver on document.body, with idempotent guards (data-enhanced, data-linked, data-act-injected). No re-entrancy or DOM-thrash risk.
  3. Keyboard shortcut isolation. Handlers no-op when focus is in INPUT, TEXTAREA, SELECT, or contentEditable. g+letter sequence times out at 1.5s.
  4. Bulk-action CSV. Generated header is the single column osm_id; values written one per line. Distinct from server-side per-zone export which carries the full cols set.
  5. JOSM remote control. Bulk-action opens http://localhost:8111/load_object?objects=w<id1>,w<id2>,…. Format matches JOSM's documented remote endpoint; no path injection vector (IDs are read from data-osm attribute set at render time).
  6. Discussion auto-link regex. /\b(way|node|relation|cs|changeset)\/(\d{2,})/gi applied to escaped HTML — no XSS surface even on raw user-typed posts.
  7. Print stylesheet. Verified via @media print that rails, dock, toolbar, console, kb-fab, and bulk bar are hidden; remaining content reflows to single-column layout.
  8. Zone bbox parity. Hard-coded zone metadata in the Atlas matches the (south, west, north, east) tuples in src/osm/zones.py for all four zones, including the corrected default.

§6Findings & recommendations

F-01
High

Discussion board has no server persistence

Threads and posts persist to localStorage only. In a multi-mapper setting this means each browser sees its own conversation. The repo has no /api/discuss route. Recommendation: add a discussions.json file alongside edit-history.json with parallel GET /api/discuss and POST /api/discuss handlers, mirroring the history pattern. Add a "Local-only" badge to the panel header until that lands.

F-02
High

Command console output is not wired to Python stdout

The console shows commands and acknowledgments but does not stream real fetch_overpass / classify log lines. The repo's logging calls in classify.py and changeset.py are silenced by sys.stdout = open(os.devnull, 'w') in server.js scan / report blocks. Recommendation: capture stderr per spawn and stream via SSE on a new /api/scan/stream; UI already has the rendering surface.

F-03
Med

Formality presets do not propagate to XLSX/dashboard generators

The formality panel exposes type-scale, voice, density, and citation toggles, but write_xlsx and write_dashboard in the repo accept no such argument today. Recommendation: add an optional formality: dict parameter to both writers and have POST /api/reports forward the panel state.

F-04
Med

Submit step does not call the OSMCha verification it advertises

The repo includes an .claude/skills/osmcha-monitor skill but no Express route. The UI presents a "verification" step after submit — that step currently resolves on its own client timer. Recommendation: add GET /api/osmcha/changeset/:id that proxies osmcha.mapbox.com with the changeset ID and returns review flags; the UI can surface true verification state.

F-05
Med

Repo skills are not exposed as actions

cagis-conflate, ground-truth-diff, maproulette-challenge, tiger-history-deep, and community-prep exist as .claude/skills/ entries but Atlas surfaces none. Recommendation: add a "Tools" sub-menu in the dock listing each skill with a one-line description; map each to a backend endpoint as it lands.

F-06
Med

"Review status / confidence" columns silent under skip_history

When the user keeps the (default) "Skip history analysis (faster)" checkbox on, those columns render blank. Today the UI does not explain why. Recommendation: render an in-cell "—" with a tooltip such as "Run scan with history analysis to populate", or auto-hide the column.

F-07
Low

Rate-limit awareness

Server enforces 100 req/min via express-rate-limit. The Atlas's command console can issue many synthetic actions in quick succession; ensure those are throttled at the client (e.g. coalesce repeated history reads). Not currently a hot path, but worth a setTimeout debouncer on the discussion auto-refresh once it goes server-side.

§7OSM-community normative checks

The repository's CLAUDE.md codifies a small set of community norms. These were verified end-to-end:

  • Mechanical-edit tagging — mechanical=yes, bot=yes, created_by, source, and a description pointing at the wiki URL are all stamped server-side at create_changeset. UI has no path to override.
  • 500-element changeset norm — enforced by CHANGESET_BATCH_SIZE; UI bulk-discuss/CSV operations cap selection rendering well under this.
  • OAuth OOB redirect — auth panel uses the documented urn:ietf:wg:oauth:2.0:oob flow.
  • User-Agent identification — osm-audit-pipeline/0.1 (Hamilton County TIGER defect audit) sent with every API call.
  • Wiki-documentation anchor — all changesets include the wiki URL; the UI does not yet link to the wiki page from its submit confirmation, which would help reviewers; minor improvement.

§8Scope & method

This audit is a code-and-DOM review. It does not exercise live OSM, Overpass, or OSMCha endpoints, and does not run the Python test suite. It compares: (a) the rendered DOM and event handlers of MetroNow Atlas.html + assets/atlas.js + assets/atlas-extras.js; against (b) the contracts declared in web/server.js, web/public/index.html, and src/osm/{zones,classify,review,changeset}.py.

Items marked "Aug" in the conformance matrix are deliberate UI augmentations beyond the repo's stated UI; they are listed so a reviewer reading web/public/index.html can see what the Atlas adds. None of them write to OSM.

§9Sign-off

Approved for continued use with the seven items in §6 tracked as follow-ups. No item blocks production; no item touches the OSM write path. The Atlas may be used to scan, review, and submit corrections today; the Discussion, Formality, and Console surfaces should be labeled "client-only" until their server contracts land.
