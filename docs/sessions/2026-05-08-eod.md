**2. `transit._increment_usage` — race condition**

Wrapped read-modify-write in `fcntl.flock(LOCK_EX)` on POSIX:

```python
try:
    import fcntl  # POSIX-only; Windows users get the unlocked path
except ImportError:
    data = _read_usage()
    data["count"] = int(data.get("count", 0)) + 1
    _write_usage(data)
    return

USAGE_FILE.parent.mkdir(parents=True, exist_ok=True)
lock_path = USAGE_FILE.with_suffix(USAGE_FILE.suffix + ".lock")
try:
    with lock_path.open("a+") as lock_fh:
        fcntl.flock(lock_fh.fileno(), fcntl.LOCK_EX)
        try:
            data = _read_usage()
            data["count"] = int(data.get("count", 0)) + 1
            _write_usage(data)
        finally:
            fcntl.flock(lock_fh.fileno(), fcntl.LOCK_UN)
except OSError as exc:
    log.warning("Could not acquire Transit usage lock: %s", exc)
    # Fall back to unlocked update — better than dropping the increment
    data = _read_usage()
    data["count"] = int(data.get("count", 0)) + 1
    _write_usage(data)
```

**Stress-tested:** 8 threads × 25 increments each, gated on a
`threading.Barrier` to maximise contention. Without the flock, the
test fails reliably. With the flock, the counter ends at exactly 200.

### 5a. On whether we've "cracked street conflation"

> **Minh:** "You're discovering some things we've had on our backburner

**Reply (after one round of "more concise please"):**

> Honestly: partly, with caveats worth naming.
>
> **What moved:** Auto-submit pool went 5.0% → 12.1% across all four

> human-review band). The single biggest lift came from switching the
> geometry term from symmetric to **directed Hausdorff (OSM→CAGIS only)**

> was punishing OSM fragments that are legitimate sub-segments of
> longer CAGIS centerlines. That insight ports to any matcher

> vibes-driven.

> **Speed limits specifically:** `set_maxspeed_cagis` is one of the
> auto-submittable fix kinds; CAGIS centerlines carry the field. The
> pipeline can fill them at scale for Hamilton County the moment we ship.
>
> **Caveats:** I haven't shipped a single changeset yet —
> everything above is "matcher output looks right," not
> community-reviewed. And the match rate is against CAGIS, a
> quarterly-maintained, survey-backed authoritative layer. The
> technique generalises; the precision depends on having a reference
> of that calibre. Counties without CAGIS-grade ground truth will get
> materially worse numbers from TIGER 2024 alone.
>
> So I wouldn't claim street conflation is cracked in general — but
> directed Hausdorff + name + direction with explicit miss-bucket
> diagnostics gets you into double-digit auto-submit against a good
> reference, without relaxing the 0.85 threshold. Happy to share the
> matcher and the F1–F4 diagnostic harness if it'd be useful for any
> of the work you've got queued.
>
> — Jared

## 6. MOTIS routing-engine prototype

**Commit:** `2b2cc3d`
**Files:** `src/osm/motis.py`, `src/osm/cli.py`, `tests/test_motis.py`,
`docs/motis-deployment.md`

### Why MOTIS

BRouter is car-only. It can't see what happens to a SORTA bus rider
on a corridor when an `oneway` tag flips. MOTIS ingests OSM and GTFS
into the same graph, so transit-corridor fixes (the ones
`bus_routes.py` marks `transit_corridor=true`) get scored against an
itinerary that includes the bus leg, not just the car detour. For
the bulk maxspeed/oneway workload BRouter is faster and produces
equivalent outcomes — this is targeted at the high-leverage minority.

### What ships

```python
# src/osm/motis.py

def fetch_route(
    origin: tuple[float, float],
    destination: tuple[float, float],
    *,
    mode: str = "WALK",
    when: dt.datetime | None = None,
    timeout: int = 30,
    use_cache: bool = True,
) -> dict | None:
    """One MOTIS /api/v5/plan call.

    Returns {length_m, duration_s, cost, geometry} matching
    osm.route_diff.fetch_route, so a future swap is one dispatcher
    line. Origin/destination are (lon, lat); MOTIS expects "lat,lon"
    so the order is flipped at the wire.
    """

def decode_polyline(encoded: str, *, precision: int = 6) -> list[tuple[float, float]]:
    """Decode a Google encoded polyline. MOTIS v5 uses precision=6
    (10× finer than Google's precision=5 default)."""

def is_available(timeout: int = 5) -> bool:
    """True if a MOTIS instance answers /api/v5/plan at MOTIS_BASE.
    Used by the CLI to decide whether to engage MOTIS."""
```

### `osm motis-status [--probe / --no-probe]`

Prints `MOTIS_BASE` and (default on) sends a one-shot probe. Exits 1
on probe failure so it's safe as a script pre-condition.

### Smoke output (no MOTIS deployed)

```
MOTIS base URL: http://localhost:8080
  (default; set MOTIS_BASE to override)
Probing /api/v5/plan…
INFO osm.motis: MOTIS health probe failed: connection refused
  → MOTIS not reachable. The pipeline will continue using BRouter;
    see docs/motis-deployment.md for setup.
Exit: 1
```

### What's NOT in the prototype (preserved verbatim from
`docs/motis-deployment.md`)

- No engine-dispatcher in `route_diff.py`. The MOTIS client returns
  the same shape as `osm.route_diff.fetch_route` so the swap is
  one-line, but the swap itself is deferred until there's a
  side-by-side comparison harness on a representative fixture set.
- No MOTIS-specific perturbation strategy. BRouter's `nogos`
  mechanism doesn't have a direct MOTIS equivalent;
  pre-edit-vs-post-edit must be done by re-ingesting the modified
  .pbf, which takes minutes.
- No bundled deployment automation. Manual binary stand-up is the
  reference; productionising it (systemd unit, restart-on-ingest
  hook, healthcheck loop) is left to the operator.

**19 new tests**, all using mocked HTTP — no live MOTIS required.
The polyline decoder test pins precision-5 vs precision-6 by ratio
(10× ratio between same-string decodings) rather than by hand-encoded
fixture, which would have been fragile.

### #3 — `auth.py:120` "clear-text logging of sensitive information"

**Triaged for "won't fix / false positive" UI dismissal.**

The print is the OAuth 2.0 + PKCE authorization URL, which by RFC
6749 §4.1.1 contains:

- `client_id` — public (it's the app identifier)
- `state` — single-use CSRF token
- `code_challenge` — hash, not the verifier (verifier stays in memory)
- `redirect_uri` — public

No actual secret. The `client_secret` stays on the server. The URL
must remain human-pasteable for the OOB redirect flow.

Dismissal comment to post in the UI: cite RFC 6749 §4.1.1 +
RFC 7636 §4.2 (PKCE).
